NeuSAP-Samples veröffentlicht „joule-a2a-agent-toolkit“ – Claude-Code-Plugin baut Joule-Agenten in Minuten

  • Pünktlich zur Sapphire hat das offizielle SAP-Samples-Repository ein neues Plugin veröffentlicht: joule-a2a-agent-toolkit – ein Claude-Code-Plugin, mit dem sich AI-Agenten für SAP Joule über das A2A-Protokoll auf BTP Cloud Foundry bauen, deployen und verbinden lassen. Für ABAP-Kollegen, die heute noch über Joule-Skills-Limits diskutieren, ist das ein konkreter Sprung in die nächste Stufe.
  • Was das Plugin liefert: Vier auto-getriggerte Skills (joule-a2a-agent, btp-cli, cf-cli, joule-cli) und drei Slash-Commands (create-agent, create-destination, deploy-agent). Mit einem Befehl wird ein vollständiger LangGraph-Agent inklusive A2A-SDK, CF-Manifest/MTA-Deployment und Joule-Capability-YAML scaffolded – in TypeScript (Express oder CAP) oder Python.
  • Stack: A2A v0.3.0 (JSON-RPC 2.0 über HTTPS), @a2a-js/sdk 0.3.10, LangGraph(.js), SAP GenAI Hub via @sap-ai-sdk/langchain OrchestrationClient. Für CAP-Agenten ist MTA-Deploy vorgesehen, für Express- und Python-Agenten der klassische cf push.
  • Beispiel-Prompt im README: „Build me a Joule A2A agent in TypeScript using CAP that checks purchase order status from S/4HANA, deploy it to eu10“ – Claude Code übernimmt Scaffolding, Destination-Anlage und Joule-Capability-Deployment in einer Pipeline.
  • Für ABAP-Architekten: Das Plugin ist die direkte Antwort auf die Frage „Wie verbinde ich meinen On-Premise-S/4-Agent mit Joule, ohne sechs Wochen BTP-Setup?“ – es ist Apache-2.0-lizenziert, läuft mit Claude Code lokal und liefert produktionsreife CF-Boilerplate. Voraussetzungen: cf CLI v8, Joule CLI (@sap/joule-studio-cli) mit App2App-IAS, Node 24, optional Python 3.12.

NeuA2A-Protokoll v1.2 unter Linux-Foundation-Governance – SAP ist Production-Member

  • Gleichzeitig zur Sapphire-Keynote hat die Linux Foundation Agentic AI Foundation bekanntgegeben, dass das ursprünglich von Google entwickelte A2A-Protokoll v1.2 jetzt unter offizieller Linux-Foundation-Governance steht und mit kryptographisch signierten Agent-Cards (Domain-Verifikation per X.509) ausgeliefert wird.
  • SAP ist Production-Member: Zusammen mit Microsoft, AWS, Salesforce und ServiceNow nutzt SAP A2A in produktiven Umgebungen – der Reach hat 150 Organisationen in Production erreicht. Damit ist A2A für SAP-Architekten kein Forschungsprotokoll mehr, sondern die strategisch gesetzte Standardschnittstelle zwischen Joule und externer Agent-Welt.
  • Konkrete Auswirkung für das joule-a2a-agent-toolkit: Das in Sektion 1 genannte Claude-Code-Plugin nutzt A2A v0.3.0 – v1.2 ist abwärtskompatibel, fokussiert aber auf signierte Identitäten. Wer jetzt Agenten baut, sollte beim Capability-Naming explizit auf das joule.ext-Namespace bestehen, um spätere Migration sauber zu halten.
  • Für Compliance-Teams: Mit Linux-Foundation-Governance ist A2A jetzt vertragsfähig – die Diskussion „darf SAP einen Google-Standard für unsere Agenten nutzen?“ ist damit beendet.
